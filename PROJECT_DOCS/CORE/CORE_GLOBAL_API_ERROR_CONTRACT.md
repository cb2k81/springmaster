# Core Global API Error Contract

## Status

Accepted as runtime Core reference with patch `000116_springmaster_global_api_error_contract_core`.

## Purpose

The Springmaster System-Core owns the reusable public API error envelope for generated and reference backend applications. Before this patch CatalogItem demonstrated the shape through demo-local DTOs and local controller exception handlers. That was sufficient as candidate evidence, but not sufficient as a Master-/Template standard.

Patch `000116` moves the baseline implementation into `de.cocondo.system.http`. CatalogItem migration is intentionally deferred to a follow-up demo-scope patch because the patch system separates Core and Demo scopes.

## Runtime types

```text
de.cocondo.system.http.ApiErrorResponse
de.cocondo.system.http.ApiViolationDTO
de.cocondo.system.http.ApiErrorType
de.cocondo.system.http.ApiErrorIdGenerator
de.cocondo.system.http.GlobalApiExceptionHandler
de.cocondo.system.exception.ResourceNotFoundException
```

`EntityAlreadyExistsException` remains in `de.cocondo.system.exception` and now carries a stable default `messageKey`.

## Canonical response shape

The global handler emits the standard Springmaster error envelope:

```json
{
  "timestamp": "2026-07-14T00:00:00Z",
  "status": 400,
  "error": "Bad Request",
  "errorType": "VALIDATION_FAILED",
  "message": "Validation failed",
  "messageKey": "springmaster.validation.failed",
  "errorId": "err-opaque-id",
  "path": "/api/demo/catalog/items",
  "method": "POST",
  "violations": []
}
```

`correlationId`, `traceId` and `localMessage` remain optional public fields. `errorId` is always generated by `ApiErrorIdGenerator` and must not encode sensitive data.

## Baseline mappings

| Source exception | HTTP status | `errorType` |
|---|---:|---|
| `MethodArgumentNotValidException` | `400` | `VALIDATION_FAILED` |
| `ConstraintViolationException` | `400` | `VALIDATION_FAILED` |
| `ValidationException` | `400` | `VALIDATION_FAILED` |
| `HttpMessageNotReadableException` | `400` | `INVALID_REQUEST` |
| `MethodArgumentTypeMismatchException` | `400` | `INVALID_REQUEST` |
| `MissingServletRequestParameterException` | `400` | `INVALID_REQUEST` |
| `IllegalArgumentException` | `400` | `INVALID_REQUEST` |
| `ResourceNotFoundException` | `404` | `RESOURCE_NOT_FOUND` |
| `EntityAlreadyExistsException` | `409` | `CONFLICT` |
| `SecurityException` | `403` | `FORBIDDEN` |
| `Exception` fallback | `500` | `INTERNAL_ERROR` |

## Safety rules

The default handler must not expose stack traces, SQL messages, framework class names or Java exception class names in fallback responses. Expected business and boundary exceptions may expose their safe domain message. Unexpected exceptions always return `Internal server error`.

## CatalogItem migration boundary

CatalogItem remains the intended first runtime consumer, but the migration is not part of this Core-scope patch. The follow-up Demo patch must remove controller-local error DTOs and local `@ExceptionHandler` methods, then prove that the public JSON error shape remains stable.
